# AGENTS.md

## 基本方針

- ユーザーから明示的な指定がない限り、日本語で回答する。
- 回答は簡潔かつ直接的にし、不要な前置きや重複を避ける。
- タスク内容が不明確な場合は、実行前に確認を取る。
- ユーザーが求めていないことを推測して実施しない。依頼された内容のみ対応する。
- 大きな変更や設計影響のある変更を行う前には、変更方針・影響範囲・実装案を整理して提示する。

---

## セーフティ境界

- 明示的な指示がない限り、基本は読み取り専用とする。
- ただし、ユーザーが「実装」「修正」「対応」「反映」など、明確な開発依頼を行った場合は、必要な範囲のファイル変更を許可されたものとして扱う。
- 削除・上書き・外部 API 書き込みなど、不可逆操作前には必ず確認を取る。
- APIキー、Token、認証情報などの機密情報は出力しない。
- DB構造変更、設定変更、大規模リファクタリングなど、影響範囲が大きい変更は事前確認を行う。

---

## 設計方針

- 保守性・機能拡張性を意識し、既存の設計方針や責務分離から外れない実装を行う。
- 機能追加時は、既存構成・責務分担・命名ルールとの整合性を確認する。
- 実装前に、既存設計に沿った実装方法を優先的に検討する。
- デザインパターンや既存フレームワークの利用は検討してよいが、必要性が明確な場合のみ採用する。
- 単純な処理に対して、不要な抽象化・過度な汎用化・「将来のためだけ」の設計を行わない。
- 「今必要な複雑さ」に留め、YAGNI（You Aren’t Gonna Need It）の考え方を意識する。
- 保守性や拡張性向上のため、既存FW・ライブラリ・デザインパターンの活用余地は検討する。ただしコストに見合わない場合は採用しない。
- 実装後、今回の変更が既存設計方針から逸脱していないか見直す。

---

## 大規模変更の定義

以下に該当する場合は「大規模変更」とみなし、実装前に変更方針・影響範囲・代替案を整理して提示する。

### 設計影響

- 責務分離やレイヤ構造への影響
- ディレクトリ構成変更
- 共通基盤・共通処理の変更
- デザインパターン導入
- 大規模リファクタリング

### データ影響

- DB構造変更
- API I/F変更
- 型定義やデータ構造変更
- 保存データ互換性への影響

### UI/UX影響

- 複数画面に影響するUI変更
- レイアウト基盤変更
- PC/スマホ両方へ影響する変更

### 変更量

- 5ファイル以上にまたがる変更
- 既存ファイルの大幅変更（目安：150行以上）
- 新規クラス追加自体は対象外。ただし責務変更を伴う場合は対象。

---

## 実装ルール

- 過度な設計を避け、依頼内容または明確に必要な変更のみ行う。
- シンプルな実装を優先する。
- 指示されていない機能追加、不要なリファクタリング、過剰な最適化は行わない。
- 変更していないコードに対して、不要なコメント・型定義・ドキュメントコメントを追加しない。
- コメントは、ロジックが自明でない場合のみ追加する。
- 実際には発生し得ないケースへの過剰なエラーハンドリングや保険的ロジックを追加しない。
- 一度しか使わないユーティリティ関数や抽象化レイヤーは作成しない。
- 不要なコードはコメントアウトせず削除する。
- 大規模変更時は、一気に変更せず段階的に分割可能な粒度へ整理する。

---

## コード構成・サイズ目安

- 原則として、1ファイルに複数の主要責務を持たせない。
- クラスを作成する場合、原則として1クラス1ファイルとする。
- 同ファイル内でのみ使う小さな型・定数・補助関数は同一ファイルに置いてよい。
- 1関数・1メソッドは目安として100行以内に収める。
- 100行を超える場合は責務分割を検討する。
- ただし、UI定義、設定値、定数一覧、テストデータなどは例外とする。
- 行数削減だけを目的に、不自然な分割や一度しか使わない抽象化は行わない。

---

## 命名規則

- 変数名・関数名・CSSクラス名・設定名は役割が一意に分かる名前にする。
- `point`, `data`, `item`, `value`, `flag`, `temp` のような意味が広すぎる名前を避ける。
- 汎用名を使う場合は、`mapPoint`, `capturePoint`, `invoiceItem`, `isMobileLayout` のように文脈を補足する。
- 名前だけで用途が推測できることを優先する。
- 略語は一般的なもの以外使用しない。

---

## 定数管理

- マジックナンバーを避け、意味のある定数名として定義する。
- UI座標、サイズ、オフセット値、しきい値など意味を持つ値は定数化または設定化を検討する。
- 将来的に調整が発生しそうな値はハードコードを避ける。
- ただし、`0`, `1`, `-1` など処理上自明な値まで過剰に定数化しない。

---

## レビュー観点

実装完了後、以下の観点でセルフレビューを行うこと。

- 要件を満たしているか
- 既存設計方針から逸脱していないか
- 過剰設計になっていないか
- 不要な抽象化をしていないか
- 命名が曖昧になっていないか
- マジックナンバーが残っていないか
- 将来修正しやすい構造になっているか
- 既存機能への副作用がないか

---

## 開発ワークフロー（Step実装時）

### 基本進め方

- 大規模変更時は、原則として **Plan → レビュー → 実装 → 停止** のStep単位で進める。
- ユーザーから明示的に許可されるまでは、次Stepへ勝手に進まない。
- 指定されたStep以外の実装を行わない。
- 実装中に仕様不整合・設計課題・想定外の影響範囲を検知した場合は、独断実装せず停止して共有する。

### Plan Mode / Default Mode

- Plan Mode時は、設計・影響範囲・実装案整理のみ行い、実装しない。
- Default Mode時のみ実装を行う。
- Modeが不明瞭な場合は確認を取る。

### Step完了時の必須対応

Step単位の実装完了時は、必ず以下を実施する。

#### 1. テスト実行

可能な限り以下を実行する。

```bash
npm.cmd run test
npm.cmd run typecheck
```

実行不可な場合は理由を明示する。

#### 2. Git状態確認

以下を実行し、差分確認を行う。

```bash
git status --short
```

意図しない変更が含まれていないことを確認する。

#### 3. Gitコミット（必須）

ユーザーから明示的に禁止されていない限り、Step完了時に必ずコミットする。

コミット前条件：

- test成功
- typecheck成功
- 意図しない差分なし

コミットメッセージ形式：

```text
type(scope): short summary
```

例：

```text
feat(battle-monitor): step3 add monitor mode tabs
refactor(gvg): step5 inject realtime subscription
refactor(battle-monitor): step6 extract shared castle list ui
test(guild-battle): update monitor title expectations
```

原則：

- Step番号を含める
- scopeを明示する
- 何をしたか短く分かる形にする

#### 4. 完了報告（必須）

Step完了時は以下を報告する。

- 変更ファイル
- 実装内容
- 変更しなかった範囲
- test結果
- typecheck結果
- `git status --short` 結果
- コミットハッシュ
- 懸念点・Known Issue（あれば）

### 安全ルール（Step実装時）

- MVP挙動を壊さないことを最優先とする。
- 無理な共通化・過剰抽象化を避ける。
- YAGNIを意識し、「今必要な複雑さ」に留める。
- 大規模変更は、一気に行わず段階的に分割する。
- 既存ユーザー影響（保存値互換・UI回帰・モバイル表示崩れ）を常に意識する。
